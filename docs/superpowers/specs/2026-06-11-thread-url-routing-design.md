# Thread URL Routing Design

## Goal

Make every Workspace thread addressable by a stable URL in the form `/c/{routeId}` so a user can copy, open, refresh, and share a link that lands directly on the matching thread. The `/c` namespace keeps concrete conversation routes separate from top-level app routes.

## Decision

Use a separate `routeId` field instead of replacing the existing semantic `thread.id`.

- `thread.id` remains the internal data key used by mock data, tests, side-window assets, replay, run inspectors, and actions.
- `thread.routeId` is the public URL identifier.
- Seed mock threads receive fixed `routeId` values so demo URLs remain stable.
- New user-created threads receive generated 16-character lowercase alphanumeric route IDs.

The route ID format is:

```text
^[a-z0-9]{16}$
```

This is intentionally a 16-character URL token, not a full 16-byte entropy encoding. It is compact and realistic enough for the demo while avoiding business-readable thread slugs.

## Routes

Thread routes live under the conversation namespace:

```text
/
/c/{routeId}
```

Legacy root-level thread routes remain readable for compatibility:

```text
/{routeId}
```

New navigation always writes `/c/{routeId}`.

Existing product routes keep priority:

```text
/product-management-platform
/product-management-platform/products/{productId}
/product-management-platform/commodities
/product-management-platform/commodities/{commodityId}
```

`/c/{routeId}` is only treated as a thread route when the second segment matches the route ID format. Unknown route IDs return the Workspace to the new-conversation state and show a toast. A legacy single-segment `/{routeId}` path is parsed the same way, but it is not generated by new UI actions.

## Workspace Behavior

### Root Path

`/` always shows the new-conversation workspace. It does not restore the last selected thread from local storage. This keeps the homepage behavior consistent with the existing demo requirement: opening the app starts from a blank new conversation unless the user explicitly opens a thread.

### Direct Thread URL

Opening `/c/{routeId}`:

1. Finds the thread by `routeId` across all non-archived threads.
2. Selects that thread and its project.
3. Expands the owning project in the sidebar.
4. Switches the top nav to `Workspace`.
5. Renders the thread workspace.

If no matching non-archived thread exists:

1. Starts a new conversation.
2. Keeps the URL at `/`.
3. Shows `Thread 不存在或已被删除`.

### Sidebar Selection

Clicking a thread in the sidebar:

1. Selects the thread through the existing store action.
2. Pushes `/c/{routeId}` to browser history.

The browser back button can return to the prior thread URL or to `/`.

### New Conversation

Clicking new conversation:

1. Starts a new thread draft through the existing store action.
2. Pushes `/` to browser history.

### Browser History

`popstate` re-parses the current path and applies it to the store:

- `/` -> new conversation.
- `/c/{routeId}` -> selected matching thread.
- `/{routeId}` -> selected matching thread through legacy compatibility.
- product-management paths -> product-management route handling.
- unknown single-segment route ID -> new conversation and toast.

## Data Model

The seed data type becomes:

```ts
export type Thread = {
  id: string
  routeId?: string
  title: string
  lastActivity: string
  transcript?: ConversationTurn[]
  runInspector?: RunInspectorData
}
```

The runtime store type becomes:

```ts
export type DemoThread = {
  id: string
  routeId: string
  title: string
  lastActivityAt: number
  pinned: boolean
  pinnedAt: number | null
  archived: boolean
  createdAt: number
  transcript: ConversationTurn[]
  runInspector?: RunInspectorData
}
```

`routeId` is required at runtime. Missing route IDs in persisted data are repaired during seed merge and state sanitization.

## Route ID Generation

Generation uses `crypto.getRandomValues` when available:

```text
alphabet = "abcdefghijklmnopqrstuvwxyz0123456789"
length = 16
```

The generator checks all existing thread route IDs and retries on collision. A timestamp/random fallback is acceptable only for environments without `crypto.getRandomValues`.

## Persistence And Migration

Persisted user-created threads from older localStorage snapshots may not have `routeId`. Migration behavior:

- Seed threads inherit their current seed `routeId`.
- User-created threads get a generated `routeId`.
- If duplicate route IDs are detected, later duplicates are regenerated.
- Existing `selectedThreadId` remains valid because it still references `thread.id`.

The persisted store version should be incremented so users pick up the new field predictably.

## Error Handling

- Invalid path formats are ignored by thread routing and continue through existing route handling.
- A valid-looking but unknown `/c/{routeId}` redirects to `/` and shows `Thread 不存在或已被删除`.
- A valid-looking but unknown legacy `/{routeId}` behaves the same way.
- Archived threads are not routable.
- If a selected thread is archived or deleted, the app navigates back to `/`.

## Testing

Required tests:

- Seed threads have unique 16-character `routeId` values.
- Runtime store hydrates every thread with a valid unique `routeId`.
- New threads receive valid unique `routeId` values.
- Direct `/c/{routeId}` visits open the matching thread.
- Legacy direct `/{routeId}` visits still open the matching thread.
- Sidebar thread clicks update the URL.
- New conversation returns the URL to `/`.
- Browser back from a thread URL to `/` shows new conversation.
- Unknown `/{routeId}` returns to `/` and shows the missing-thread toast.
- Product-management routes continue rendering the product-management page.

## Out Of Scope

- Server-side routing or backend persistence.
- Share permissions and access control.
- Public short-link management.
- Replacing existing semantic thread IDs.
- Introducing `react-router`.
