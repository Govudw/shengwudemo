# BioMap 通用产品立项会文字稿

**版本：30分钟汇报版**
**主题：建设 Agentic AI、实验、数据与企业服务的公司级公共底座**

---

## 一、核心观点

BioMap 需要建设一套面向全公司产品矩阵的**通用产品体系**。

这套通用产品不是某一条行业线的业务产品，也不是一个单纯的研发中台，而是 BioMap 未来所有行业产品线共同复用的 **Agentic AI、实验、数据和企业服务底座**。

它要解决的核心问题是：

第一，各行业产品线都会重复需要 Agent 框架、实验系统、数据服务、企业服务等基础能力。如果每条线各自建设，会造成大量重复投入。

第二，各行业产品线的 AI 水平不一致。有的产品线可能只是对话式 AI，有的具备 RAG 能力，有的具备工具调用能力，有的希望进一步形成可规划、可执行、可追踪、可迭代的 Agentic AI。如果没有统一底座，公司的 AI 能力会分散在不同团队中，难以复用，也难以整体提升。

第三，BioMap 未来真正的产品壁垒不只是某个单点模型或某个单点应用，而是 **Agent、实验、数据、知识库、模型、用户行为和企业服务之间形成闭环**。这个闭环必须通过通用产品来沉淀。

因此，通用产品的核心价值可以概括为一句话：

> 让 BioMap 从“各产品线重复搭系统”升级为“统一底座支撑多行业产品快速扩展”，并把全公司的 Agentic AI 能力拉到同一个更高水平。

---

## 二、为什么不是各个产品线自己做

从单条产品线角度看，自建基础能力短期似乎更快。但从公司整体看，长期会带来四个问题。

### 1. 重复造轮子，研发资源被低差异化能力消耗

每条行业产品线都会需要类似能力：

- 账号登录、权限、租户、SSO、用户中心；
- Agent 框架、工具调用、Skill 管理、任务日志、评估监控；
- LIMS、ELN、实验流程、实验记录、工作流编排；
- 数据接入、知识库、文献文档、模型服务、审计追踪；
- 企业资源管理、商品管理、订单、合同、计费、业法财联通。

这些能力本质上不是某一条行业线的核心差异化能力，而是 BioMap 所有产品都会反复用到的公共能力。

如果每条产品线都自己做，一开始可能只是多做几个模块，但当产品线数量增加后，就会变成公司级的重复投入。

行业线的研发资源应该更多投入到行业 Know-how、客户场景、模型效果、业务闭环和商业化交付，而不是反复搭账号、权限、数据、Agent 框架和实验流程。

### 2. AI 能力分散，各产品线的 Agentic AI 水平不一致

这是当前最需要强调的问题。

未来 BioMap 的产品竞争力，很大程度上取决于各行业产品线能不能把 AI 从“辅助问答”升级为“能完成任务的 Agentic AI”。

但如果每条产品线独立建设 AI 能力，就会出现明显差异：

- 有的产品线只能做基础问答；
- 有的产品线能做知识库检索；
- 有的产品线能调用少量工具；
- 有的产品线能做多步骤任务执行；
- 有的产品线能连接实验、数据和模型反馈；
- 有的产品线具备日志、评估、回放和审计；
- 有的产品线完全没有这些能力。

这样带来的结果是，公司不同产品的 AI 体验、AI 可靠性、AI 可解释性和 AI 交付能力会非常不一致。

对于客户而言，他们看到的是 BioMap 整体品牌。如果某些产品线 AI 能力强，某些产品线 AI 能力弱，会影响客户对 BioMap AI 产品化能力的整体判断。

通用 Agent 的价值，就是把 Agentic AI 的核心能力沉淀为统一框架，让所有行业产品线都能站在同一个 AI 底座上起步。

它既能**抬高 AI 能力的下限**，也能**拉高 AI 能力的上限**。

### 3. 数据、实验和 AI 无法形成公司级闭环

BioMap 的长期优势不是单纯来自某个模型，而是来自持续积累的数据、实验反馈、文献知识、用户行为和模型迭代。

如果每条产品线的数据、实验流程、Agent 日志、知识库和审计记录都分散建设，就会带来几个问题：

- 数据无法统一治理；
- 知识库无法跨产品复用；
- 实验结果无法系统性反哺模型；
- Agent 行为数据无法统一评估；
- 客户操作和审计记录无法形成公司级资产；
- 未来跨行业、跨产品的协同会非常困难。

通用数据服务和通用实验平台的意义，就是把这些资产统一纳管，在权限可控、过程可追踪、使用可审计的前提下，让数据和实验真正成为公司级复利。

### 4. toB 交付难以规模化

BioMap 面向企业客户时，客户不仅看 AI 能力和业务效果，也会关注企业级基础能力：

- 是否支持 IAM；
- 是否支持 SSO；
- 是否支持组织架构；
- 是否支持多租户和资源隔离；
- 是否支持权限管理；
- 是否支持审计记录；
- 是否支持资源配额和用量统计；
- 是否支持合同、订单、计费和交付流程。

如果这些能力由各产品线分别补齐，就会导致每个客户都像项目制交付，边际成本很高，客户体验也不一致。

企业服务系统的价值，是把 toB 交付产品化、标准化，让 BioMap 具备承接更多企业客户和更复杂组织结构的能力。

## 三、通用产品的定位

通用产品的定位可以概括为：

> BioMap 全产品矩阵的公共产品底座，支撑各行业产品线快速构建 Agentic AI 产品、实验产品、数据产品和企业级 toB 服务。

它包含四大能力：

| 通用产品模块 | 核心定位 |
|---|---|
| 通用 Agent | 全公司 Agentic AI 能力底座，提供通用 Agent 核心 loop、工具调用、Skill 打包、任务执行、日志审计和评估能力 |
| 通用实验平台 | 实验管理和实验流程底座，提供 LIMS、ELN、工作流编排、原子组件接入和实验记录追踪 |
| 通用数据服务 | 公司级数据和知识底座，提供数据目录、权限、知识库、文献文档、基础模型、操作记录和审计追踪 |
| 企业服务系统 | toB 商业化和企业交付底座，提供 IAM、SSO、用户中心、资源管理、商品管理、业法财联通等能力 |

这四个模块共同支撑 BioMap 从单点产品走向产品矩阵。

## 四、通用 Agent：提升全公司 Agentic AI 水平

通用 Agent 是本次通用产品中最关键的能力之一。

它不是某一个具体 Agent，而是一个可以支撑多种 Agent 形态的通用 Agent 框架。

通过和不同模型、知识库、工具、Skill、工作流和前端形态组合，它可以快速形成不同产品线需要的 Agent。

例如：

- 文献分析 Agent；
- 实验设计 Agent；
- 数据分析 Agent；
- 项目管理 Agent；
- BD 支持 Agent；
- 客户服务 Agent；
- 合规审查 Agent；
- 企业知识问答 Agent；
- 研发流程 Agent。

这些 Agent 表面上服务不同场景，但底层都需要一套共通能力。

### 1. Agentic AI 能力可以分层理解

为了让管理层更清晰地理解“AI 水平提升”的价值，可以把 Agentic AI 能力分成几个层级。

| AI 能力层级 | 典型能力 | 产品表现 |
|---|---|---|
| L0：基础 AI 问答 | 基于大模型进行对话 | 能回答问题，但不能稳定完成任务 |
| L1：知识库增强 AI | RAG、文档检索、知识问答 | 能基于企业知识和文献回答问题 |
| L2：工具调用 AI | 调用数据库、API、模型、分析工具 | 能完成部分操作型任务 |
| L3：任务型 Agent | 任务规划、多步骤执行、过程检查、结果汇总 | 能围绕一个目标持续推进任务 |
| L4：闭环研发 Agent | 连接数据、模型、实验、反馈和审计 | 能支持研发闭环和持续优化 |

如果各产品线自己做，可能有的停留在 L0/L1，有的能到 L2，少数能到 L3，但很难形成统一的 L4 能力。

通用 Agent 的目标，就是让所有行业产品线至少具备 L2/L3 能力，并为未来走向 L4 闭环研发 Agent 打基础。

### 2. 通用 Agent 应具备的核心能力

| 能力模块 | 说明 |
|---|---|
| Agent 核心 loop | 任务理解、规划、执行、反馈、修正、终止判断 |
| 工具调用框架 | 支持调用数据库、知识库、模型、实验系统、外部 API 和内部工具 |
| Skill 打包机制 | 把常用能力封装成可复用 Skill，不同 Agent 可以组合使用 |
| 模型接入与路由 | 支持不同模型接入，根据任务类型选择合适模型 |
| 知识库接入 | 支持文献、专利、内部文档、实验记录、企业知识等检索和引用 |
| 上下文管理 | 管理任务上下文、历史记录、项目状态和用户偏好 |
| 人工确认机制 | 对关键步骤支持 human-in-the-loop，避免 Agent 越权或误操作 |
| 任务日志与回放 | 记录 Agent 每一步做了什么、调用了什么、为什么这么做 |
| 权限与审计 | 确保 Agent 访问数据、工具和实验流程时遵守组织权限 |
| 效果评估 | 评估任务成功率、工具调用成功率、人工介入率、失败原因等 |
| 前端轻适配 | 通过轻量前端改造，快速形成不同 Agent 产品形态 |

通用 Agent 的关键不是“做一个聊天机器人”，而是建设一套 BioMap 自己的 Agentic AI 运行时。

### 3. Agentic AI 水平提升的量化价值

AI 能力提升可以从几个方面量化。

| 指标 | 各产品线自建时的常见情况 | 有通用 Agent 后的目标 |
|---|---|---|
| 新 Agent 从想法到 Demo | 4-8 周 | 1-2 周 |
| 新 Agent 从 Demo 到可试点 | 2-3 个月 | 3-6 周 |
| 通用工具 / Skill 复用率 | 每条线重复开发 | 50%-70% 可复用 |
| Agent 任务成功率 | 各线差异大，难以统一评估 | 通过统一评估体系提升 10-30 个百分点 |
| 人工介入率 | 高度依赖人工判断和补救 | 在标准任务中下降 20%-40% |
| 工具调用成功率 | 各线各自调试 | 统一框架后持续优化，目标达到 80%-90% 以上 |
| Agent 问题定位周期 | 依赖研发排查日志 | 通过日志、回放、审计缩短 30%-50% |
| AI 能力一致性 | 各产品体验差异明显 | 各线共享同一套基础 Agentic AI 能力 |
| AI 能力沉淀 | Prompt、工具、经验分散在各团队 | Skill、工具、评估集、日志统一沉淀 |

这里最重要的不是某个单点指标，而是整体 AI 能力的复利。

每一条产品线做出的 Prompt、工具、Skill、评估集、Agent 日志、失败案例和最佳实践，都可以回到通用 Agent 底座中，成为其他产品线可复用的能力。

这会让 BioMap 的 AI 产品不是一个个孤立的 AI 功能，而是一个不断增强的 Agentic AI 平台。

### 4. 通用 Agent 对业务的直接价值

通用 Agent 能带来三类业务价值。

第一，**更快做出 AI 产品**。

行业线不需要从零搭 Agent 框架，而是直接使用通用 Agent 核心 loop，配置行业数据、行业 Skill、行业工具和前端形态。这样，新 Agent 从想法到 Demo 的周期可以从数周压缩到一两周。

第二，**更稳定地交付 AI 能力**。

统一 Agent 框架可以提供统一的日志、权限、工具调用、错误处理和评估体系。这样，Agent 不只是“能演示”，而是逐渐具备可监控、可回放、可迭代、可交付的产品能力。

第三，**让 AI 水平跨产品线提升**。

某一条产品线做出的工具、Skill、评估方法和知识库能力，可以被其他产品线复用。这样，AI 能力不会锁在单个团队里，而会沉淀成公司级能力。

## 五、通用实验平台：连接 AI 和真实实验闭环

通用实验平台覆盖 LIMS、ELN、工作流编排和原子组件接入。

它的目标是让各行业产品线可以更快接入实验流程、实验数据和实验服务。

| 能力模块 | 说明 |
|---|---|
| LIMS 基础能力 | 样本、试剂、耗材、批次、项目、实验任务管理 |
| ELN 基础能力 | 实验记录、实验模板、过程记录、结果归档、版本管理 |
| 工作流编排 | 支持实验 SOP、任务流转、审批、自动化节点和人工节点 |
| 原子组件接入 | 接入模型预测、数据分析、实验任务、报告生成等组件 |
| 实验数据采集 | 统一接收实验结果、附件、图片、表格和过程数据 |
| 实验权限与审计 | 控制谁能看、谁能改、谁能执行，并保留全链路记录 |
| 实验结果追踪 | 打通实验设计、实验执行、结果分析和模型反馈 |

它的价值不只是替代若干实验管理工具，而是让 AI 建议、实验执行、实验结果和模型迭代形成闭环。

未来，一个 Agent 提出实验建议后，可以通过通用实验平台进入任务流；实验结果产生后，可以进入通用数据服务；数据再反哺模型和 Agent 评估。这是 BioMap 区别于普通 AI 工具公司的重要能力。

## 六、通用数据服务：沉淀公司级数据资产

通用数据服务负责把各行业产品线的数据模块、基础模型、知识库、文献文档、实验数据、平台操作记录和审计记录，在严格权限控制下统一沉淀和服务。

| 能力模块 | 说明 |
|---|---|
| 数据目录 | 管理数据资产在哪里、是什么、属于谁、能否使用 |
| 元数据管理 | 管理数据类型、来源、版本、标签和关联关系 |
| 数据权限 | 支持用户、组织、项目、租户和数据级权限控制 |
| 数据血缘 | 记录数据来源、加工过程、被哪些模型和 Agent 使用过 |
| 知识库服务 | 支持文献、专利、内部文档、实验记录和业务知识统一检索 |
| 文献文档服务 | 支持导入、解析、结构化、摘要、问答和引用追踪 |
| 基础模型服务 | 管理各产品线共用的基础模型、嵌入模型和推理服务 |
| 操作与审计记录 | 记录用户、Agent、系统对数据和工具的访问与操作 |
| 数据 API 服务 | 为各行业产品线提供统一、权限可控、可追踪的数据调用能力 |

通用数据服务的重点不是简单把数据集中存放，而是让数据在权限可控、来源可追踪、使用可审计的前提下，真正被多个产品线复用。

这会让 BioMap 的数据从“项目资产”升级为“公司资产”。

## 七、企业服务系统：支撑 toB 规模化商业化

企业服务系统是 BioMap toB 业务的公共商业化底座。

它分为前台企业服务能力和后台经营支撑能力。

### 1. 前台企业服务能力

| 能力模块 | 说明 |
|---|---|
| 企业 IAM | 企业身份管理、组织架构、成员管理 |
| SSO | 支持企业统一登录和身份集成 |
| 用户中心 | 统一账号资料、认证方式、账号状态 |
| 租户管理 | 支持不同企业、项目和环境隔离 |
| 权限管理 | 支持角色、组织、项目、资源和数据多层权限 |
| 资源管理 | 管理算力、模型调用、数据空间、实验资源、配额和用量 |
| 企业审计 | 为客户管理员提供操作记录、访问记录和安全审计 |

### 2. 后台经营支撑能力

| 能力模块 | 说明 |
|---|---|
| 产品商品管理 | 管理产品包、功能包、服务包、价格和售卖规则 |
| 合同订单联通 | 打通销售、合同、订单、交付和续费流程 |
| 计费与用量 | 支持按用户、资源、调用量、项目、服务周期等方式计费 |
| 业法财联通 | 连接业务、法务、财务流程，减少线下协作和人工对账 |
| 客户交付管理 | 支持客户开通、配置、授权、变更、停用和续约 |
| 内部运营后台 | 支持运营、交付、财务、法务、管理员等角色使用 |

企业服务系统的核心价值，是让 BioMap 从项目制、手工化交付，逐步走向产品化、标准化、规模化交付。

## 八、通用产品的整体价值

通用产品的价值可以分为五类：降本、提效、提升 AI 水平、支撑扩张、沉淀数据飞轮。

### 1. 降本：减少重复研发投入

假设未来 BioMap 有 4-6 条行业产品线，每条产品线如果独立建设基础能力，都会重复投入类似模块。

| 通用能力 | 单条产品线自建投入估算 | 重复建设风险 |
|---|---:|---|
| Agent 框架、工具调用、Skill 管理、日志评估 | 4-8 人月 | AI 能力不一致，Agent 经验无法复用 |
| LIMS/ELN/实验流程基础模块 | 10-20 人月 | 实验数据结构割裂，流程无法统一 |
| 数据接入、知识库、权限、血缘、审计 | 8-15 人月 | 数据资产无法复用，治理成本升高 |
| IAM、SSO、用户中心、租户权限 | 6-10 人月 | 企业交付重复建设，客户体验不一致 |
| 资源管理、商品管理、订单 / 计费联通 | 5-10 人月 | 商业化流程难以规模化 |

按这个口径估算，单条产品线基础能力投入约为 33-63 人月。

如果有 4 条产品线，重复投入约为 132-252 人月。

如果有 6 条产品线，重复投入约为 198-378 人月。

通用产品统一建设后，虽然平台本身也需要投入，但随着产品线数量增加，边际复用价值会明显提升。一个合理目标是：在 2 年周期内，让公共能力相关重复研发投入减少 30%-50%。

### 2. 提效：缩短新产品、新 Agent、新客户上线周期

| 场景 | 没有通用产品 | 有通用产品后的目标 |
|---|---:|---:|
| 新 Agent 从概念到 Demo | 4-8 周 | 1-2 周 |
| 新 Agent 从 Demo 到客户试点 | 2-3 个月 | 3-6 周 |
| 新行业产品基础底座搭建 | 3-6 个月 | 4-8 周 |
| 新实验流程上线 | 4-10 周 | 1-3 周 |
| 新数据源接入并进入权限治理 | 2-6 周 | 3-10 天 |
| 新企业客户基础开通 | 4-8 周 | 1-2 周 |

这个价值对公司非常重要。

AI 和生命科学结合的市场机会变化很快，客户试点窗口也经常是阶段性的。谁能更快做出 Demo、更快交付试点、更快形成数据闭环，谁就更容易占据客户心智。

### 3. 提升 AI 水平：让各行业产品线共享 Agentic AI 能力

这是通用产品最具战略意义的价值之一。

各行业产品线如果各自建设 AI 能力，AI 水平会逐渐分化。强的产品线会越来越强，弱的产品线可能长期停留在基础 AI 问答阶段。

通用 Agent 可以把 AI 能力拆成一组可复用模块：

- 任务规划；
- 工具调用；
- Skill 封装；
- 模型路由；
- 知识库检索；
- 上下文管理；
- 人工确认；
- 任务日志；
- 权限审计；
- 效果评估；
- 失败回放；
- 持续优化。

这些能力一旦沉淀为统一底座，每条行业线都可以直接复用。

这意味着，通用 Agent 不仅是在节省研发成本，更是在提升全公司的 AI 产品能力。

可以设定以下衡量指标：

| 指标 | 目标 |
|---|---:|
| 通用 Agent 框架覆盖的产品线数量 | 覆盖主要行业产品线 |
| 新 Agent 中复用通用 Skill 和工具的比例 | 50%-70% |
| 标准任务 Agent 成功率提升 | 10-30 个百分点 |
| 标准任务人工介入率下降 | 20%-40% |
| Agent 工具调用成功率 | 80%-90% 以上 |
| Agent 问题定位周期缩短 | 30%-50% |
| Agent 日志和审计覆盖率 | 接近 100% |
| 评估集、Prompt、Skill 复用次数 | 持续增长 |

对公司来说，这些指标背后代表的是一个更大的变化：

BioMap 不是每条线各自做 AI 功能，而是在形成统一的 Agentic AI 能力平台。

### 4. 支撑扩张：让新行业线不再从零开始

未来 BioMap 进入新的行业方向、新客户场景、新 Agent 形态时，如果每次都要重新搭系统，扩张速度会被底层建设拖慢。

通用产品可以把新产品建设方式变成：

- 统一 Agent 框架 + 行业 Skill；
- 统一实验平台 + 行业实验流程；
- 统一数据服务 + 行业数据资产；
- 统一企业服务 + 行业商业化方案。

这样，新行业线不再从零开始，而是在统一底座上快速配置和扩展。

### 5. 沉淀飞轮：形成 Agent、实验、数据、模型闭环

通用产品的长期价值，是帮助 BioMap 形成产品飞轮。

这个飞轮可以这样理解：

> Agent 帮助用户完成任务；任务过程中调用知识库、模型、工具和实验系统；实验和业务结果进入数据服务；数据服务沉淀为结构化资产和知识；这些数据反过来优化模型、知识库、Agent Skill 和评估体系；优化后的能力继续服务更多产品线和客户。

这个闭环一旦形成，BioMap 的产品能力会随着使用量、客户数、实验数和数据量增长而持续增强。

这才是通用产品最长期的价值。

它不是简单降本，而是帮助 BioMap 形成平台化扩张能力。

## 九、Agent Demo 穿插建议

这次立项会建议加入 Agent Demo。

Demo 的作用不是单纯展示一个功能，而是让老板和高层直观看到：

- 我们做的不是一个单点 Agent；
- 我们做的是一个可复用的 Agentic AI 框架；
- 同一套 Agent 核心 loop，可以通过不同 Skill、工具、数据和前端形态，快速变成不同产品线的 Agent。

### Demo 建议放在汇报中段

建议在讲完“为什么 AI 能力必须通用化”之后立刻进入 Demo。

这样效果最好。

因为前面已经讲清楚了问题：各产品线 AI 水平不一致、Agent 能力重复建设、AI 能力无法沉淀。

然后用 Demo 证明：通用 Agent 可以解决这个问题。

### Demo 讲解重点

Demo 可以控制在 5-7 分钟。

不要把 Demo 讲成某个具体业务功能的演示，而要围绕“通用能力”来讲。

建议 Demo 讲解结构如下：

#### 1. 先说明 Demo 重点

可以这样讲：

> 接下来这个 Demo，重点不是展示某一个具体 Agent，而是展示通用 Agent 底座的能力。
> 我们希望让大家看到，一个 Agent 从接收任务、理解任务、规划步骤、调用工具、使用知识库、生成结果、记录日志，到后续评估和审计，背后可以由同一套通用框架支撑。
> 未来不同产品线只需要替换行业数据、Skill、工具和前端形态，就可以快速形成自己的 Agent。

#### 2. 展示 Agent 接收任务

可以选择一个和 BioMap 业务贴近、但不需要过度解释行业细节的任务。

例如：

- “基于一组文献和内部知识，生成一个实验设计建议。”
- “分析某个研发问题，检索相关文献，调用内部模型，生成下一步实验方案。”
- “针对某个项目目标，拆解任务、调用数据、生成分析报告。”

重点是让领导看到 Agent 不是简单问答，而是在完成一个多步骤任务。

#### 3. 展示 Agent 规划过程

展示 Agent 如何把任务拆成几个步骤：

- 理解目标；
- 检索知识库；
- 调用工具或模型；
- 整理结果；
- 生成建议；
- 标记需要人工确认的环节。

这一部分要突出 Agentic AI 和普通 Chatbot 的区别。

普通 Chatbot 是回答问题。

Agentic AI 是围绕目标规划和执行任务。

#### 4. 展示工具 / Skill 调用

展示 Agent 调用某个工具、模型、知识库或分析组件。

这里要强调：

- 这个工具不是写死在某个 Agent 里的；
- 它可以被封装成 Skill；
- 这个 Skill 未来可以被多个 Agent 复用；
- 一个产品线沉淀的 Skill，可以被其他产品线复用。

#### 5. 展示日志、权限和审计

如果 Demo 已经具备日志或过程记录，建议一定展示。

这对高层非常重要，因为它说明这个 Agent 不是“黑盒 AI 玩具”，而是有机会成为企业级产品能力。

可以强调：

- Agent 访问了哪些数据；
- 调用了哪些工具；
- 每一步做了什么；
- 是否需要人工确认；
- 是否有审计记录；
- 后续能否回放和评估。

#### 6. Demo 结束后的收束话术

Demo 结束后可以这样收束：

> 这个 Demo 背后真正重要的，不是这个 Agent 本身，而是这套通用 Agent 框架。
> 如果没有通用 Agent，每条产品线都要自己做任务规划、工具调用、Skill 管理、权限、日志和评估。
> 有了通用 Agent 之后，不同产品线可以共享同一套 Agentic AI 能力，只是在行业数据、工具、Skill 和前端体验上做差异化。
> 这就是通用产品对 BioMap AI 水平提升的核心价值。

## 十二、30 分钟现场讲稿精简版

各位今天我想汇报的是 BioMap 通用产品立项。

这件事的核心不是多做一个内部平台，而是建设 BioMap 未来产品矩阵的公共底座。

BioMap 未来会有多条行业产品线，每条产品线都会需要 Agent、实验、数据和企业服务能力。如果每条线都自己建设，短期看起来快，但长期会出现重复建设、AI 能力不一致、数据实验割裂、toB 交付不标准的问题。

其中我认为最关键的是 AI 能力问题。

现在各行业产品线对 AI 的使用水平可能是不一样的。有的还停留在基础问答，有的能做知识库检索，有的能调用工具，有的希望做多步骤任务型 Agent。未来真正有价值的 AI 产品，不只是一个 Chatbot，而是能理解目标、规划步骤、调用工具、使用数据、连接实验、记录过程、接受审计，并持续优化的 Agentic AI。

如果每条产品线各自建设 AI 能力，公司的 AI 水平会分散在不同团队里，难以复用，也难以整体提升。

所以我们需要通用 Agent。

通用 Agent 不是某一个具体 Agent，而是一套 Agentic AI 核心框架。它包括任务理解、规划执行、工具调用、Skill 打包、上下文管理、权限控制、日志审计和效果评估。不同产品线可以在这套框架上配置自己的行业数据、行业工具、行业 Skill 和前端形态，快速形成自己的 Agent。

这样带来的价值有两个。

第一，抬高下限。所有行业产品线都不需要从基础 AI 能力开始搭，可以直接站在统一 Agent 底座上。

第二，拉高上限。通过统一连接数据、实验、模型和知识库，未来可以形成真正的闭环研发 Agent。

除了通用 Agent，我们还需要通用实验平台、通用数据服务和企业服务系统。

通用实验平台提供 LIMS、ELN、工作流编排和原子组件接入能力，让实验流程、实验记录、实验结果可以统一追踪。

通用数据服务负责数据目录、知识库、文献文档、基础模型、数据权限、血缘和审计，让各行业产品线的数据在权限可控的前提下成为公司级资产。

企业服务系统负责 IAM、SSO、用户中心、资源管理、商品管理、订单计费和业法财联通，让 BioMap 的 toB 交付从项目制走向产品化和规模化。

从量化价值上看，通用产品可以带来几个方面的收益。

在研发投入上，公共能力重复建设预计可以减少 30%-50%。

在上线速度上，新 Agent 从概念到 Demo 可以从 4-8 周缩短到 1-2 周；新行业产品底座搭建可以从 3-6 个月缩短到 4-8 周；新企业客户开通可以从 4-8 周缩短到 1-2 周。

在 AI 能力上，通用 Skill 和工具复用率可以达到 50%-70%；标准任务 Agent 成功率有机会提升 10-30 个百分点；人工介入率在标准任务中可以下降 20%-40%。

更重要的是，通用产品会帮助 BioMap 形成一个长期飞轮。

Agent 帮助用户完成任务，过程中调用知识库、模型、工具和实验平台；实验和业务结果进入数据服务；数据服务反哺模型、知识库、Skill 和 Agent 评估；优化后的能力再服务更多产品线和客户。

这就是通用产品的长期价值。

它不是简单降本，而是帮助 BioMap 形成平台化扩张能力。

接下来通过一个 Agent Demo，可以更直观看到这个逻辑。这个 Demo 重点不是某一个 Agent 能做什么，而是它背后的通用能力：任务规划、工具调用、Skill 配置、知识库接入、日志追踪和未来可复用能力。
