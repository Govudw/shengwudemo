Core Thesis 02 / 17

BioMap 需要一套给所有产品线共用的产品底座，并按明确节点推进到可交付状态。

这次要做的，是把通用 Agentic AI、智能实验平台、数据资产管理、企业服务前台和后台系统做成各产品线都能复用的公共能力。
少重复账号、权限、Agent、实验、数据这些基础能力，不要每条线都做一遍。
上线更快新 Agent、新产品、新实验流程、新企业客户都可以少走一段基础建设。
AI 能力一致让各产品线至少共享任务规划、工具调用、日志、评估这些 Agentic AI 能力。
方便扩张多条产品线在同一套底座上配置和交付，不再各自补基础系统。
最终是让新行业产品和企业客户交付更快、更标准、更可复制。
核心判断别让每条产品线都重复搭一遍基础系统
Core Product Scope 03 / 17

核心功能不是一个单独平台，而是五类公司级公共能力。

通用 Agentic AIAgent loop、工具调用、Skill、任务日志、权限审计、评估与回放。优先通过行业 Agent 场景验证。
智能实验平台LIMS、ELN、实验工作流、原子组件接入、实验结果追踪。连接 AI 建议和真实实验闭环。

企业服务前台系统审批、权限管理、通知、用户中心、费用中心和企业管理员入口。支撑企业客户可见服务。
企业服务后台系统产商品管理、账单系统、合同订单联通、运营配置和交付管理。短期 SOP / 表格化过渡。
功能边界通用产品聚焦中后台、通用框架和公共能力

Why Now 05 / 17

现在必须做，是因为多条产品线已经开始重复要 Agent、权限、数据接入和企业交付能力。

1
多行业产品线并行

每条线都会反复需要账号、权限、数据、Agent、实验和交付能力。

2
Agent 需求变多

文献分析、实验设计、项目管理、BD 支持、合规审查，底层能力其实是一套。

3
实验、数据、模型要连起来

AI 建议、实验执行、实验结果、模型反馈不能停留在分散系统里。

4
企业客户要求更高

IAM、SSO、多租户、审计、资源配额、合同订单计费是规模化 toB 的基本门槛。

5
各自建设会越做越散

产品线越多，重复投入、体验不一致、数据难复用的问题越重。

已出现

这些需求已经不只是单个产品线的问题；再晚统一，后面就是系统整合成本。

现在先把公共能力收住，后续产品线扩张才不会反复补基础系统。

为什么现在重复需求已经出现，越晚统一越贵
Self-built Risk 06 / 17

各产品线自建，短期快，长期会变成五类重复成本。

各线自建对象
会带来的问题
Agent 框架各自建设

实验平台各自建设

数据服务各自建设

核心判断这些通用能力不该反复占用行业线研发
Agentic AI Maturity 07 / 17

这里最要紧的是：各产品线的 Agentic AI 水平会拉开差距。

L0基础问答能回答问题，但不能稳定完成任务。
L1知识库增强基于文献、内部文档和企业知识回答问题。
L2工具调用调用数据库、API、模型、分析工具完成部分操作。
L3任务型 Agent围绕目标进行规划、执行、检查、汇总和修正。
L4闭环研发 Agent连接数据、模型、实验、反馈和审计，支持持续优化。
如果各线自建：能力可能停留在 L0/L1/L2，少数到 L3，难以统一到 L4。
通用 Agent 先把主要产品线带到 L2/L3，再为 L4 闭环研发 Agent 做准备。
没有通用 Agent，各线最多各自做到工具调用；有公共底座，任务规划、日志评估和审计才能公司级复用。
AI 能力定位先保证下限，再往 L4 走

Industry Linkage 09 / 17

先接入行业 Agent 场景，验证通用能力，再沉淀复用组件。

01
用一个行业 Agent 场景跑通公共能力选择真实、高频、可验证的任务，跑通任务理解、规划、工具调用、数据接入、权限审计、日志评估和结果输出。
02
将可复用部分沉淀回通用产品沉淀前端小组件、行业 Skill、工具接入模板、数据接口配置、权限审计规则、评估样例和交付经验。
03
按 Alpha / Beta / GA 管理交付节奏Alpha 验证链路；Beta 进入真实试点；GA 成为行业产品可稳定依赖的标准能力。
Agent loop 和通用框架由通用产品统一定义不在每条行业线里反复定义，行业线重点提供真实任务、行业数据和验收标准。
中后台能力原则上接入所有行业产品数据资产管理系统、企业服务前台和后台系统不作为独立行业场景入口，也不单独对外验证。
复用对象要具体复用 Skill、工具接入、前端组件、数据接口和审计规则，不把“复用”停留在口号上。
联动机制行业产品线给真实场景，通用产品沉淀公共能力
Customer Security 10 / 17

客户使用的全流程安全，要覆盖登录、授权、执行、审计和回放。

01身份进入企业 IAM、SSO、用户中心确认人、组织和租户。
02访问授权按角色、组织、项目、resource 和数据边界授权。
03任务执行Agent 调用工具、数据和实验流程前，先做权限校验。
04过程留痕记录用户、Agent、系统对资源和工具的访问与操作。
05审计回放按用户、资源、动作和时间追踪问题与责任边界。
当前权限控制可以细到 resource 级。
审批：资源申请、访问授权和高风险变更可进入审批。
通知：权限变更、任务状态和安全事件及时触达。
边界：Agent 不能绕过企业前台系统直接访问资源。
安全边界安全能力属于企业服务前台系统的一部分

Agent Demo Moment 13 / 17

DEMO展示

01接收任务基于文献和内部知识，生成实验设计建议。
02理解目标识别项目目标、约束、输入材料和需要确认的边界。
03规划步骤拆成检索、模型调用、分析、建议生成和人工确认。
04调用能力使用知识库、内部模型、数据分析工具或实验系统。
05生成结果输出方案、证据、风险点和下一步实验建议。
06记录过程留下日志、权限、审计、回放和评估数据。
Demo 要证明三件事：一个任务能跑完、能力能复用、过程可审计。
一个任务能跑完：从目标理解到结果生成，不停在问答。
能力能复用：核心 loop、Skill 和工具能给其他 Agent 用。
过程可审计：日志、权限、回放和评估数据都能留下来。
Demo 判断标准任务完成、能力复用、过程可审计

Company Support 16 / 17

当前不需要额外人力，重点需要公司确认优先级、试点场景和商业化边界。

公共底座定位通用产品服务多条行业产品线，不按单一项目需求建设。
行业 Agent 试点行业线提供真实、高频、可验证的场景、数据和验收标准。
中后台系统接入数据资产、企业服务前台和后台原则上接入各行业产品。
权限安全审计当前可做到 resource 级权限控制；后续统一 Agent 调用工具、访问数据、触发流程的边界原则。
商业化口径Agentic AI 先作为行业属性智能体的一部分出售；未达到 GA 前，不做规模商业化。
后台短期过渡企业服务后台短期用 SOP 和表格化方式支撑，Q4 再产品化。
资源诉求需要的是协同和边界，不是新增人力
Closing 17 / 17

建议立项：按 2026 年节点，把五类通用能力做成行业产品共用底座。

01

核心功能是五类：通用 Agentic AI、智能实验平台、数据资产管理、企业服务前台和后台系统。

02

行业联动先从行业 Agent 场景切入，把 Skill、组件、数据接口和审计经验沉淀回通用产品。

03

当前不需要新增人力；需要确认公共底座定位、中后台接入、resource 级权限审计原则和商业化边界。
